## 1. 배포 환경의 변화: Manual vs CI/CD
**From Docker/Nginx to GitHub Actions**

* **초기 접근:** GitHub Actions의 기능을 알지 못해, 개인 NAS 서버에서 Docker와 Nginx를 활용해 수동으로 배포를 진행했습니다. Bridge 네트워크와 리버스 프록시(Reverse Proxy)를 설정하여 개인 도메인으로 연결했습니다.
* **배운 점:** GitHub Actions를 통한 **자동 빌드 및 배포(CI/CD)**의 존재를 알게 되었으며, 이를 통해 반복적이고 번거로운 수동 배포 과정을 자동화할 수 있음을 깨달았습니다.

## 2. CI/CD 도입 후 겪은 문제점과 워크플로우 재정립
GitHub Actions를 도입했으나, 실제 운영 과정에서 몇 가지 한계에 부딪혔습니다.

* **Issue:**
    * 배포 속도가 예상보다 느리고, 간헐적으로 변경 사항이 즉시 반영되지 않는 현상(캐시 혹은 배포 지연) 발생.
    * 워크플로우가 멈추거나(Cancel 불가), HTML 수정 후 리빌드 시에도 이전 코드로 롤백되는 등 불안정한 상황 경험.
* **Insight:**
    * 빌드-배포의 자동화 자체는 매끄럽고 편리했으나, **개발 및 테스트 단계**까지 의존하기엔 비효율적임을 확인했습니다.
    * **결론:** 사양 수정 및 기능 테스트는 로컬(또는 개인 NAS) 환경에서 완벽히 검증한 뒤, 검증된 코드만 Main 브랜치로 푸시하여 배포하는 것이 가장 안정적인 워크플로우임을 확립했습니다.

## 3. 코드 구조의 이해: 기능 구현을 넘어 구조 설계로
**JSP에서 순수 HTML/CSS/JS로의 회귀와 성장**

* **기존의 방식:** 이전의 JSP 기반 프로젝트(마이페이지, 채팅, 웹소켓 등)는 '기능 구현'에만 치중하여 코드의 가독성과 구조가 매우 취약했습니다.
* **변화와 성장:**
    * 이번 프로젝트를 통해 기본 마크업(HTML)과 스타일(CSS)의 문법을 재정립하고, **가독성 있는 코드 구조**의 중요성을 체감했습니다.
    * **조립의 즐거움:** 마치 제가 좋아하는 취미인 **프라모델이나 레고를 조립하듯**, 모듈 단위로 코드를 쌓아 올리며 전체적인 사이트 구조를 개편해 나가는 개발의 즐거움을 느꼈습니다.
